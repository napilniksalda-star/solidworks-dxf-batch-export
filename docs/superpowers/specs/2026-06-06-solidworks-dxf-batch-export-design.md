# Пакетный экспорт развёрток листового металла в DXF — обновление макроса

- **Дата:** 2026-06-06
- **Файл-исходник:** `Macro1.swp` (модуль `Macro11`, SolidWorks VBA)
- **Тип:** доработка существующего макроса
- **Вариант реализации:** Вариант 2 («надёжный»)

---

## 1. Контекст и цель

Есть рабочий макрос SolidWorks, который для открытой детали-пластины проходит по всем
конфигурациям и выгружает развёртку листового металла в DXF (для лазера/раскроя).
Имена файлов формируются по размерам и количеству.

Заказчик присылает таблицу заказа (пример — `ЗАКАЗ 46 - для чертежей.xlsx`), где для каждой
позиции указаны размеры и количество. Оператор вручную переносит данные в **таблицу параметров**
(design table) детали — каждая строка становится конфигурацией — и запускает макрос.

**Цель доработки:**
1. Добавить **диалог выбора папки** для сохранения DXF.
2. Сделать так, чтобы корректно менялись **длина, ширина и толщина** (раньше менялась только длина).
3. Починить скрытые проблемы старого макроса (см. §2).

## 2. Предыстория и корневая проблема

Старый макрос читал длину/ширину/количество из **пользовательских свойств** (`Length`, `Width`,
`Quantity`). Свойство к геометрии напрямую не привязано. В прошлом заказе все пластины были
шириной 52 мм, и свойство `Width` фактически оставалось статичным — поэтому **во всех файлах была
одна ширина 52**, независимо от реальной геометрии. Это и есть корневая проблема: имя файла не
отражало фактическую пластину.

В заказе №46 уже встречаются разные ширины (52, 96, 176) и толщины (1,5 и 2,0), поэтому полагаться
на статичные свойства нельзя.

**Решение:** брать размеры **из геометрии**, а не из свойств.

## 3. Источники данных (per configuration)

| Параметр | Источник | Кто заполняет в таблице параметров |
|---|---|---|
| Длина | размер `D1@Эскиз1` (читается из модели) | оператор (столбец размера) |
| Ширина | размер `D2@Эскиз1` (читается из модели) | оператор (столбец размера) |
| Толщина | элемент **Sheet-Metal** (читается из модели) | оператор задаёт размер толщины; макрос читает сам |
| Количество | свойство `Quantity` (`$PRP@Quantity`) | оператор (один столбец-свойство) |

- Длина и ширина читаются как значения размеров: `swModel.Parameter("D1@Эскиз1").SystemValue * 1000`
  (SystemValue в метрах → мм), с округлением. Не зависит от единиц документа.
- Толщина читается из определения листового металла активной конфигурации
  (`SheetMetalFeatureData.Thickness * 1000`), имя размера толщины знать не нужно.
- Количество читается из свойства конфигурации (`CustomPropertyManager.Get4`), т.к. в геометрии его нет.

## 4. Функциональные требования

1. Запускается на открытой, сохранённой детали (`part`).
2. Спрашивает у пользователя папку для DXF (по умолчанию — папка детали).
3. Проходит по **всем конфигурациям** детали.
4. Для каждой конфигурации, где есть листовой металл/развёртка, выгружает DXF развёртки.
5. Имя файла отражает **реальные** длину × ширину × толщину и количество.
6. Не перезаписывает файлы при совпадении имён (строки-дубли в заказе).
7. Возвращает деталь на исходную активную конфигурацию.
8. Показывает итоговый отчёт: сколько выгружено, сколько с ошибкой, список проблемных конфигураций.

## 5. Логика работы (по шагам)

1. **Проверки входа:** есть `ActiveDoc`; это `swDocPART`; деталь сохранена (есть путь).
   Иначе — сообщение и выход.
2. **Выбор папки:** `Shell.Application.BrowseForFolder`, стартовая папка — папка детали.
   Отмена → выход (сообщение «Отменено»).
3. **Запомнить** имя текущей активной конфигурации.
4. **Цикл** по `swModel.GetConfigurationNames`:
   1. `ShowConfiguration2(confName)` + `EditRebuild3`.
   2. Прочитать длину (`D1@Эскиз1`), ширину (`D2@Эскиз1`) из размеров → мм, округлить.
   3. Прочитать толщину из элемента Sheet-Metal → мм, округлить.
      Если листового металла нет → пропустить конфигурацию, записать в отчёт «нет листового металла».
   4. Прочитать `Quantity` из свойства (если пусто → `NA`).
   5. Сформировать базовое имя: `Plate_{Длина}x{Ширина}x{Толщина}_Qty_{Кол-во}.dxf`.
   6. **Защита от перезаписи:** если файл существует — добавить суффикс имени конфигурации;
      если и так существует — добавить счётчик `_2`, `_3`, …
   7. Экспорт развёртки: `swPart.ExportToDWG2(..., swExportToDWG_ExportSheetMetal, ...)`.
   8. Учесть результат (успех/ошибка) для отчёта.
5. **Восстановить** исходную активную конфигурацию (`ShowConfiguration2`).
6. **Итоговый отчёт:** `MsgBox` «Готово! Выгружено N из M. Ошибок: K» + детали в Immediate
   (`Debug.Print`) по проблемным конфигурациям.

## 6. Имя файла

Шаблон: `Plate_{Длина}x{Ширина}x{Толщина}_Qty_{Количество}.dxf`

Примеры: `Plate_200x52x1.5_Qty_2.dxf`, `Plate_50x176x2_Qty_150.dxf`

Правила форматирования чисел:
- разделитель дробной части — **точка** (`1.5`), не запятая;
- незначащие нули убираются (`2.0 → 2`, `1.50 → 1.5`);
- округление до 2 знаков после запятой;
- размеры длины/ширины обычно целые, но форматируются по тем же правилам.

## 7. Обработка ошибок и краевые случаи

| Ситуация | Поведение |
|---|---|
| Нет открытого документа / не деталь / не сохранён | Сообщение, выход (как сейчас) |
| Отмена диалога выбора папки | Сообщение «Отменено», выход |
| В конфигурации нет листового металла / развёртки | Пропуск + запись в отчёт |
| Размер `D1@Эскиз1`/`D2@Эскиз1` не найден | Пропуск конфигурации + запись в отчёт |
| Свойство `Quantity` пустое/отсутствует | В имени `Qty_NA`, экспорт выполняется |
| `ExportToDWG2` вернул `False` | Запись в отчёт как ошибка |
| Совпадение имени файла | Суффикс конфигурации, затем счётчик |

## 8. Настройки (константы вверху макроса)

Для адаптации без правки логики:
- `LENGTH_DIM = "D1@Эскиз1"`
- `WIDTH_DIM  = "D2@Эскиз1"`
- `QTY_PROP   = "Quantity"`
- `FILE_PREFIX = "Plate_"`

## 9. Вне рамок (YAGNI)

- **Прямое чтение Excel-заказа** — оператор по-прежнему переносит данные в таблицу параметров вручную.
  (Возможная отдельная задача в будущем.)
- **Форма (UserForm)** — без неё; диалог выбора папки — стандартный системный.
- **Раскладка DXF по подпапкам толщины** (`3мм/`, `5мм/`) — не делаем (это был Вариант 3).
- **Материал** — в имени и логике не учитываем.

## 10. Критерии готовности

- На детали с несколькими конфигурациями (разные длина/ширина/толщина) макрос создаёт по одному
  DXF на каждую валидную конфигурацию в выбранной папке.
- Имена отражают **реальные** размеры из геометрии (баг «вечная ширина 52» отсутствует).
- Файлы с одинаковыми размерами не перезаписывают друг друга.
- После выполнения деталь остаётся на исходной конфигурации.
- В конце показан отчёт со счётчиками и списком проблем.

## 11. Допущения

- Пластины — из листового металла (есть элемент Sheet-Metal и развёртка); часть деталей гнётся
  после резки, но DXF — это плоская развёртка (прямоугольник), поэтому толщина на контур не влияет
  и нужна только для имени.
- В каждой детали длина = `D1@Эскиз1`, ширина = `D2@Эскиз1` (вынесено в константы — можно поменять).
- Толщину листового металла в активной конфигурации можно надёжно прочитать из
  `SheetMetalFeatureData.Thickness` после активации и перестроения конфигурации. Если для каких-то
  деталей это окажется не так — резервный вариант: читать размер толщины по имени (уточняется на этапе реализации).
- UI-сообщения — на русском (как в текущем макросе).
